# Генератор коммерческих предложений

MVP веб-приложения для генерации КП в DOCX и PDF по DOCX-шаблонам. Первый шаблон сидируется из файла `КП.docx`: приложение создает из него рабочую копию с плейсхолдерами и сохраняет как версию шаблона «КП».

## Что реализовано

- Google-вход через Google Identity Services и проверку backend-ом.
- Белый список email, роли `admin` и `manager`, блокировка пользователей.
- История КП: менеджер видит только свои КП, администратор видит все.
- Создание, редактирование, дублирование и удаление КП.
- Ручной ввод адресата и товаров, включая email адресата при наличии.
- Импорт реестра товаров из XLSX и автозаполнение наименования по реестровому номеру.
- Автоматический расчет строк, итога, НДС 22%, суммы прописью на русском.
- Исходящий номер формата `ДДММ/ручная-часть/М`.
- PDF-предпросмотр по кнопке.
- Генерация DOCX и PDF одной операцией, отдельные скачивания.
- Версионирование шаблонов на уровне БД.
- Хранение только последних финальных DOCX/PDF.
- Автоудаление КП через 100 календарных дней при старте приложения, предупреждение за 5 дней в списке.
- Базовый аудит действий.

## Первый запуск

1. Скопируйте `.env.example` в `.env`.
2. Заполните `APP_SECRET`, `GOOGLE_CLIENT_ID`, `SEED_ADMIN_EMAIL`.
3. Убедитесь, что `КП от Кристины.docx` лежит в корне проекта.
4. Запустите:

```bash
docker compose up --build
```

Приложение будет доступно на `http://SERVER_IP:8000`.

## Локальный dev-вход

Для проверки без Google можно включить:

```env
DEV_LOGIN_ENABLED=true
SEED_ADMIN_EMAIL=admin@example.com
```

После старта войдите через форму локального входа с `admin@example.com`.

## Реестр товаров

Администратор может загрузить реестр товаров в разделе `Админ-панель` → `Реестр товаров`.

Файл должен быть в формате `.xlsx`. Приложение читает первый лист:

- колонка `A` — полное наименование товара;
- колонка `D` — реестровый номер;
- пустые строки и строки без наименования или номера пропускаются;
- строки-заголовки пропускаются, если значение в колонке `D` не состоит из цифр;
- реестровый номер хранится как строка;
- повторная загрузка обновляет существующие записи без создания дублей.

В форме КП в строке товара можно указать `Реестровый номер`. Если номер найден, поле `Наименование` автоматически заполняется в формате:

```text
Наименование товара
Реестровый номер: XXXXXXXX
```

В итоговом DOCX/PDF отдельная колонка для реестрового номера не создается: обе строки выводятся внутри колонки `Наименование товара`.

## Подготовка реального шаблона

Сейчас приложение автоматически подготавливает шаблон из `КП от Кристины.docx`, заменяя конкретные данные на плейсхолдеры и сохраняя стили документа. Для максимально надежного совпадения лучше вручную подготовить DOCX-шаблон с плейсхолдерами:

- `{{outgoing_number}}`, `{{quote_date}}`
- `{{recipient_name}}`, `{{recipient_inn}}`, `{{recipient_email}}`, `{{recipient_address}}`
- `{{intro_text}}`, `{{specification_text}}`, `{{optional_conditions}}`
- `{{item_no}}`, `{{item_name}}`, `{{item_unit}}`, `{{item_quantity}}`, `{{item_unit_price}}`, `{{item_line_total}}`
- `{{total_amount}}`, `{{vat_rate}}`, `{{vat_amount}}`
- `{{total_amount_words}}`, `{{vat_amount_words}}`
- `{{delivery_term}}`, `{{warranty}}`, `{{valid_until}}`

Строка таблицы с `{{item_*}}` используется как шаблон строки товара. Ее стиль копируется для всех позиций.

## Важные ограничения MVP

- Для PDF нужен LibreOffice. В Dockerfile он установлен.
- Полное визуальное совпадение зависит от качества исходного DOCX-шаблона и установленных шрифтов на сервере.
- Автоудаление запускается при старте приложения. Для production стоит добавить отдельный scheduled worker/cron.
- Админская загрузка шаблонов есть, но тонкая настройка схемы плейсхолдеров оставлена на следующий этап.
